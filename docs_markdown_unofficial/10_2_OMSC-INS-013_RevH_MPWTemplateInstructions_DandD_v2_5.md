GENERAL TEMPLATE INSTRUCTIONS

This document, Mission Package Worksheet (MPW) Template Instructions,
OMSC-INS-013, provides information on how to fill in required
information in the Mission Package Worksheet (MPW) Template,
OMSC-TMP-006. Sections 1 and 2 of this MPW Template Instructions are
aligned with Sections 1 and 2 of the MPW Template, while Section 3 of
this document provides guidance on how to fill in the MPW table in the
MPW Template. The green text in each section of this MPW Template
Instructions should be used as guidance to fill out the corresponding
section in the MPW Template. Green text in the MPW Template Instructions
is considered guidance only and should be removed to reflect your MPW.
NOTE: The first page of the MPW Template is for Open Mission Systems
(OMS) purposes only and should be removed prior to the release of the
completed MPW.

The MPW Template Instructions also identifies version and revision
history for the template itself, versus that of an MPW derived from the
template. When creating an MPW, the last sentence in its
Abstract/Summary, identifying the revision and date of the MPW Template
to which the MPW conforms, should match the version information found
herein for this matching MPW Template Instructions. The abstract is a
brief summary of the document that follows and is used to help the
reader quickly ascertain document’s purpose and overall content.

A Mission Package provider should save the MPW Template using an
appropriate unique filename. The Mission Package provider should add an
appropriate document number, revision history and document date for
configuration management of program specific information. The
distribution statement should be filled out based on the program
specific restrictions, and at a minimum should be Distribution Statement
A. If the contents of the document require a higher restriction, the
appropriate distribution statement should be updated as needed. Green
text in the MPW Template represents guidance for the creation of the
MPW. Green text should not remain in the completed MPW and should be
replaced in the cover page, Abstract, Revision Record, and all
applicable sections defined in the MPW Template.

Each OMS Mission Package requires an associated MPW to be compliant. The
OMS MPW documents key implementation details of an OMS Mission Package.
When an Adopting Program (AP) decides to incorporate a compliant OMS
Mission Package into a weapon system, the Mission Package Worksheet
(MPW) is used to identify and provide key top-level information on the
Mission Package Units of Replaceability; viz., the OMS Mission Package,
Platform, Services, Subsystems, and Isolators of the Mission Package, as
applicable. The MPW complements the MPDD, by describing top-level
information on the UoRs, while the MPDD describes the “as built”
security policies and data configuration of an OMS Mission Package.
These will evolve during the life-cycle of an OMS Mission Package’s
development.

This document contains only non-proprietary information. Classification
of the MPW should be based on the contents of the document.

Modify the sections in the template using the following guidelines:

1.  **Existing black text is required and should not be deleted.**

2.  **If a section is not applicable**, keep that section heading and
    add “Not Applicable” with a rationale for why that section is not
    applicable on a new line. For figures and tables below the not
    applicable section, replace figures with “Figure not applicable” and
    enter “N/A” in the first empty cell for tables.

The rationale behind not deleting table captions, figure captions, or
sections is to preserve section references and table/figure identifiers
to maintain traceability within the document and between documents. For
example, the Security Control Traceability Matrix (SCTM) traces a
security control to a specific document template, section and/or table
identifier.

1.  **Maintain existing table column headers.**

Tables can be further modified (e.g., adding additional columns when
applicable) but the original structure of existing tables should be
preserved.

1.  **Delete all green guidance text in the section, including green
    text added in table rows provided for exemplar purposes only.**

2.  **If additional tables or figures are added**, the table/figure
    caption must follow the third level numbering system. Existing,
    default captions already in the template are numbered against
    headings at level 2 (e.g., under Section 1, the first table in the
    section is numbered Table 1.0-1. Under Section 2.4.1, the second
    table in the section is numbered Table 2.4-2). Newly added
    tables/figures should be numbered against headings at level 3 (e.g.,
    under Section 1, the first newly added table in the section is
    numbered Table 1.0.0-1. Under Section 2.4.1, the second newly added
    table in the section is numbered 2.4.1-2). This results in
    preserving the numbering of the original tables.

Ensure correct table and figure caption styles are applied so they
appear in the List of Tables and List of Figures. Use the “TableCaption”
Style for table captions and “FigureCaption” Style for figure captions.

All documents referenced within the MPW must include a document
reference number with release dates and be included in the compliance
package deliverable.

The documents listed in Table 0.0-1, OMS D&D Documents, are the
documents that assist in the understanding and implementation of OMS.

Overview
========

The purpose of the MPW is to identify the Units of Replaceability (UoRs)
composing an OMS Mission Package during different phases of program
development. OMS defines UoRs as the ‘building blocks’ of a Mission
Package that can be separately and independently replaced and/or
upgraded, specifically to include the Mission Package itself and its
constituent Platform, Subsystems, Services, and Isolators. The MPW
should identify each OMS Service, Subsystem, and Isolator in a Mission
Package. Each completed row of the MPW identifies one of the Mission
Package’s OMS UoRs. Table 1.0-1 depicts the MPW.

This document is intended to assist the reader in creating a Mission
Package Worksheet (MPW) by properly filling in the information required
by the Mission Package Worksheet (MPW) Template, OMSC-TMP-006.

In the table below, the ‘green text’ entry references the section number
in this document which provides guidance on completing the cell.

When an Adopting Program (AP) decides to incorporate a compliant OMS
Mission Package into a weapon system, the MPW is used to document the
planned UoRs of that Mission Package. MPWs can be used to document UoRs
both within a specific Mission Package as well as across a portfolio of
systems with OMS Mission Packages. Therefore stakeholders of a MPW can
include both those supporting the Adopting Program (AP) of a specific
OMS system/Mission Package as well as higher level stakeholders with
goals and concern spanning multiple OMS systems. Design decisions
documented in the MPW should reflect the goals of all applicable
stakeholders. The information specified by the MPW template may also be
included as part of an acquisition document (Request for Information,
Request for Proposal, or a proposal).

Initially the MPW documents planned and/or assumed details of the
Mission Package, and as the design, implementation, and integration of
the Mission package evolves, the MPW is updated. The MPW allows the AP
to continuously assess Mission Package content and determine if the
Mission Package is meeting the intent and requirements of the AP.

Speculative MPWs can be used to document and evaluate proposed changes
to a Mission Package, and earlier versions of the MPW can be used to
document the evolution of a Mission Package across its life cycle.

When a Mission Package is completed, the MPW serves as a standardized
top-level summary of the Mission Package composition and is a key
artifact used to assess the OMS compliance of the Mission Package and
its Platform, Subsystems, and Services. The MPW identifies the
compliance Tier for each of the applicable UoRs in the Mission Package,
and identifies which Mission Package Service Capabilities and Subsystems
are on the ‘Lists for Compliance’ defining compliance requirements for
compliance tiers 1 and 2. Note that a compliance tier is not identified
for an Isolator, as Isolators take on the compliance tier of the Mission
Package.

Finally, the MPW identifies the Cyber Survivability Risk Category (CSRC)
of the Mission Package.

Mission Package Worksheet (MPW) Guidance 
========================================

Mission Package Worksheet Header and Footer
-------------------------------------------

The contents of the MPW Header and Footer distribution statement should
be filled out based on the specific classification, identification
numbering, and distribution restrictions of the Adopting program (AP).

### Header

The MPW document header should include the following information:

-   Classification – A classification marking IAW program-specific
    guidance. The originator should reference their appropriate Security
    Classification Guide(s).

-   Document Number – The document number assigned to the MPW

-   Revision – The applicable revision identifier.

### Footer

The MPW document footer should include the following information:

-   A distribution statement/reminder – The distribution statement (for
    the title page) or the distribution reminder (for subsequent pages)
    should be filled out based on the program specific restrictions, and
    at a minimum should be Distribution Statement A. If the contents of
    the MPW require a different distribution restriction, the
    appropriate distribution statement and footer reference should be
    updated as needed.as applicable per the AP sponsoring the Mission
    Package.

-   Classification – A classification marking which matches the Header

-   Page Number – A page number on every page.

OMS Mission Package Section
---------------------------

This section of the MPW lists the Mission Package name, acquisition
phase, OMS Standard version, and compliance tier, and Cyber
Survivability Risk Category (CSRC).

### Entry Instructions for ‘Mission Package Name’

This field should identify the name of the Mission Package. This name
should be sufficient to uniquely identify the Mission Package.

### Entry Instructions for ‘Phase’

This field should identify the point in the acquisition process that
this version of the MPW was targeted. For example, this could be “Draft
RFP,” “RFP,” “Proposal,” “PDR,” “CDR,” or “Final,” etc. While these
examples are typical, a Program may have other points at which the MPW
is generated, so program-unique identifiers can be used here. For
example: “iPDR” or “dPDR” or “PDR2,” etc.

### Entry Instructions for ‘OMS Standard Version’

This field should identify the version of the OMS Standard applicable to
the Mission Package being documented by the MPW. For example, this could
be “1.0” or “2.1.” Associated with this version of the OMS Standard is
the OMS schema version that applies as well.

### Entry Instructions for ‘Tier’

This field should identify the compliance tier of the Mission Package,
and be one (1) of:

-   “Tier 1”

-   “Tier 2”

-   “Tier 3”.

If the MPW is being completed prior to the implementation and compliance
assessment of the Mission Package, this field should identify the
Mission Package’s required/desired compliance tier. If the MPW is being
completed after implementation and compliance assessment, this field
should identify the Mission Package’s achieved compliance tier.

The compliance Tier of a Mission Package is a roll-up based on the Tier
levels of the Mission Package components as documented within the rest
of the worksheet. Essentially it is the minimum value of the Tier levels
for all of the tiered components as identified in the Tier field for
each tiered component. For example, if the OMS Platform is Tier 2, all
listed Services are Tier 2, and the single Subsystem is Tier 1, then the
Mission Package Tier level is Tier 1. Compliance is determined based on
the requirements, verification requirements, acceptance criteria and
compliance checklists of the specific OMS Standard version indicated on
the MPW form. Note that the OMS Message Schema version is implicitly
included here by reference.

Note that the objective Mission Package compliance Tier may be based on
the needs of the specific AP incorporating the Mission Package, or may
have been assigned at a higher level in the case of a Mission Package
used, or intended to be used, by multiple APs.

### Entry Instructions for ‘CSRC’

This field should identify the CSRC assigned to the Mission Package, and
is a value ranging from one (1) to five (5). The cybersecurity posture
as defined by the CSRC and described by the CSAs denotes what an OMS
Architecture Element achieves as one element within a cyber contested
environment. The CSRC is typically assigned by the program.

If this OMS Mission Package does not have a program-assigned
Cybersecurity requirement, “N/A” is a valid assignment in the CSRC
field.

Note: The OMSC-GDE-003, Cybersecurity Guide, defines concepts and
guidance for OMS Adopting Programs and system designers to utilize for
documenting the OMS Architecture Element cybersecurity posture.

Note that the objective Mission Package CSRC may be based on the needs
of the specific AP incorporating the Mission Package, or may have been
assigned a higher level in the case of a Mission Package used, or
intended to be used, by multiple APs.

OMS Platform Section
--------------------

This section of the MPW lists the Platform name, UCI Schema Version(s),
and compliance tier.

### Entry Instructions for ‘Platform Name’

This field should identify the name of the OMS Platform.

### Entry Instructions for ‘UCI Schema Version(s)’

This field should identify the UCI version, or versions, used as the
base schema (i.e., before application of extension schema(s)) supported
by the Platform CAL(s). Note that while a UCI Schema is associated with
the OMS Standard version identified in the Mission Package section,
Platform CALs may be based on multiple UCI Schema versions.

### Entry Instructions for ‘Tier’

This field should identify the compliance tier level of the Platform,
and should be one of:

-   “Tier 1”

-   “Tier 2”

-   “Tier 3”.

If the MPW is being completed prior to the implementation and compliance
assessment of the Platform, this field should identify the Platform’s
required/desired compliance tier. If the MPW is being completed after
implementation and compliance assessment, this field should identify the
Platform’s achieved compliance tier.

The compliance Tier of an OMS Platform is assessed per OMSC-CHK-003, OMS
Platform Checklist.

Note that the objective Platform compliance Tier may be based on the
needs of the specific Mission Package and/or AP incorporating the
Platform, or may have been assigned at a higher level in the case of a
Platform used, or intended to be used, by multiple APs.

OMS Service Section
-------------------

This section lists the Service name, Service Capability List category,
Service Location, and Tier for each Service that is part of the Mission
Package, including OMS Services hosted in the Platform Open Computing
Environment (OCE) or other Platform mission processing, as well as OMS
Services in OMS Subsystems included in the OMS Mission Package. There
should be a row in this section for each OMS Service in the Mission
Package.

### Entry Instructions for ‘Service Name’

This field should identify the name of the Service. For example, this
could be “Fusion,” “Route Planner,” or “Sensor Manager.”

### Entry Instructions for ‘Service Capability List Category’

This field should identify the Service’s category using the options from
the “Service Capability List for Compliance” section of the OMS Standard
document (OMSC-STD-001). If a Service listed in this section of the MPW
does not match or align with any of the capabilities in the Service
Capability List for Compliance, an entry of “Not Applicable,” should be
used.

### Entry Instructions for ‘Service Location’

This field should identify where the Service is located inside the
Mission Package. This field should be one (1) of:

-   “OCE”

-   The OMS Subsystem name that is hosting the Service

-   “Other mission processing”.

### Entry Instructions for ‘Tier’

This field should identify the compliance tier level of the Service, and
should be one (1) of:

-   “Tier 1”

-   “Tier 2”

-   “Tier 3”.

If the MPW is being completed prior to the implementation and compliance
assessment of the Service, this field should identify the Service’s
required/desired compliance tier. If the MPW is being completed after
implementation and compliance assessment, this field should identify the
Service’s achieved compliance tier.

The compliance Tier of an OMS Service is assessed per OMSC-CHK-005, OMS
Service Checklist.

Note that the objective Service compliance Tier may be based on the
needs of the specific Subsystem, Mission Package, and/or AP
incorporating the Service, or may have been assigned at a higher level
in the case of a Service used, or intended to be used, by multiple APs.

OMS Subsystem Section
---------------------

This section lists the Subsystem name, Subsystem List category, CAL
Location, and Tier for each Subsystem that is part of the Mission
Package. There should be a row in this section for each OMS Subsystem in
the Mission Package.

### Entry Instructions for ‘Subsystem Name’

This field should identify the name of the specific Subsystem. For
example, this could be “AN/APY-XX,” “AN/SLQ-YY,” or “Electro Optical
(EO) System.”

### Entry Instructions for ‘Subsystem Capability List Category’

This field should identify the Subsystem’s category using the options
from the “Subsystem List for Compliance” section of the OMS Standard
document (OMSC-STD-001). If a Subsystem listed in this section of the
MPW does not match or align with any of the categories in the “Subsystem
List for Compliance”, an entry of “Not Applicable,” should be used.

### Entry Instructions for ‘CAL Location’

This field should identify the location of the CAL used by the Subsystem
or its OMS Adapter. This field should be one (1) of:

-   “OCE” – For the case where the CAL is located in the OCE

-   “Subsystem” – For the case where the CAL is located in the Subsystem
    hardware configuration

-   “Other Mission Processing” – For the case where the CAL is located
    in mission processing of the Platform outside of the OCE.

The location of the CAL used by the Subsystem or its Adapter is
typically either in the specific Subsystem or inside mission processing,
although it is possible that a Subsystem Adapter could run in the OCE.
This functionality includes reporting status and providing responses to
specific Subsystem commands. This Adapter does not need to be listed in
the Service Compliance section of the MPW as it is considered to be an
integral part of the Subsystem.

### Entry Instructions for ‘Tier’

This field should identify the compliance tier level of the Subsystem,
and should be one of:

-   “Tier 1”

-   “Tier 2”

-   “Tier 3”.

OMS Isolators Section
---------------------

This section identifies and lists the names, non-OMS endpoints, and
Isolator location for each Isolator that is part of the Mission Package.
There should be a row in this section for each OMS Isolator that uses
the CAL.

For example, Satellite Communication (SATCOM) could be considered an
external interface, and thus, might have an Isolator to send messages
to/from the OMS Mission Package. Another example of an Isolator is the
interface to a Vehicle Management System (VMS) that provides position
information or time synchronization information or both to the Mission
Package.

### Entry Instructions for ‘Isolator Name’

This field should identify the name of the Isolator. For example, this
could be “VMS Bridge,” “SATCOM Gateway,” or “Flight Display Bridge.” The
name does not have to contain the term Isolator, although that is
recommended.

### Entry Instructions for ‘Non-OMS Endpoint’

This field should identify the non-OMS endpoint of the Isolator. For
example, this could be “NAV System,” “Flight Management System,” or
“Flight Displays.”

### Entry Instructions for ‘Isolator Location’

This field should identify the location of the Isolator. This field
should be one of:

-   “OCE”

-   “Other mission processing”.
