Introduction
============

This list of acronyms and glossary of terms provides definitions for key
abbreviations and terms used throughout the OMS Standard documents.

Open Mission Systems (OMS) is a non-proprietary, open architecture
standard for integrating subsystems and services into mission packages.

The acronyms and abbreviations are those used throughout the documents
comprising the OMS Definition and Documentation (D&D) Standard. The
glossary was built to include terms that might have multiple meanings or
in some way, the meaning is customized for use in OMS.

References
==========

The documents listed in Table 2.0-1, OMS D&D Documents, are the
documents that assist in the understanding and implementation of OMS.
The documents listed in Table 2.0-2, Other Documents, are cited
throughout this document. For dated references, only the edition cited
applies. For undated references, the latest edition of the referenced
document (including any amendments or corrigenda) applies.

<td>The Abstract Service Bus (ASB) is a composite element that supports all types of OMS Data Exchanges. The ASB includes the physical layer that is the foundation of the ASB’s other sub-elements. </tr>

<td>Removes all classified data created during Subsystem operation, returning “the Subsystem to its default/native/base classification level”. Depending on a Subsystem’s design, CDE can be commanded as a part of the Subsystem shutdown command (probably the normal use), or could be invoked during normal Subsystem operation. A Subsystem will document its support for CDE in its SDD and/or Service Contract, and should reject a CDE command for situations it does not support. Restoring power to a Subsystem shutdown with CDE restarts the Subsystem to normal operation. See also potentially related terms: ZEROIZE and DISABLE.</td>
</tr>
<tr class="even">

However, data exchanges that require very low latency performance may choose to utilize OMS Data Transfer or OMS Special Signal Data Exchanges.</td>
</tr>

<td>Critical Abstraction Layer (CAL)</td>
<td>A Critical Abstraction Layer (CAL) is an OMS architecture concept and approach supporting the standardized OMS Messaging between the OMS Platform and the Subsystems, Services, and Isolators that are part of an OMS Mission Package. A CAL is realized by a CAL Implementation that supports the CAL API used by Subsystems, Services, and Isolators. The CAL isolates Subsystem, Service, and Isolator application software from Platform specific CAL Implementations. A Platform will include one or more CAL Implementations that allow Subsystem, Services, and Isolators to communicate using OMS Messages. The CAL interface aids portability of Subsystem, Services, and Isolators among OMS Mission Packages. The CAL allows applications to interact (get, set, send, receive, etc.) with message objects regardless of the underlying object implementation.</td>
</tr>
<tr class="odd">

</tr>
<tr class="even">
<td>Critical Abstraction Layer (CAL) Implementation</td>

</tr>

<td>Isolation</td>
